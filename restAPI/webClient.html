RestTemplate vs WebClient

Understanding Blocking vs Non-Blocking Communication in Spring Boot Microservices

What is the Main Difference?

RestTemplate

RestTemplate is a traditional synchronous HTTP client in Spring Boot.

Blocking Nature ❌

When a thread sends a request, it waits until the response comes back.

During waiting time:

  • Thread becomes busy
  • Cannot handle other requests
  • Wastes server resources

WebClient

WebClient is a reactive non-blocking HTTP client introduced in Spring WebFlux.

Non-Blocking Nature ✅

Thread sends request and immediately becomes FREE.

It does NOT wait for response.

  • Better scalability
  • Less thread usage
  • High performance

Problem with RestTemplate

Suppose:

  • Client sends 1 request
  • Your API internally calls 3 microservices

If you use RestTemplate, the main thread waits for all responses.

Problem ❌

Main thread is blocked while waiting for APIs.

If thousands of users come:

  • Huge thread consumption
  • Memory increases
  • CPU context switching increases
  • Server becomes slow

Still There is a Problem ⚠️

Even though APIs are parallel:

  • Main thread still exists
  • 3 additional worker threads created

Total threads used:

4 Threads Reserved

If 10,000 requests come: 

10,000 × 4 = 40,000 threads

Huge scalability problem.

How WebClient Solves This

WebClient uses:

  • Event Loop
  • Reactive Streams
  • Non-Blocking I/O
  • OS-level socket notification
What Happens Internally ✅
  • Thread sends request
  • Thread becomes FREE immediately
  • OS watches sockets using epoll/kqueue
  • OS informs event loop when response comes
  • Event loop executes callback

Why WebClient is Faster?

Less Threads

Few event-loop threads can handle thousands of requests.

No Blocking

Threads are never stuck waiting for API responses.

High Scalability

Perfect for microservices and high traffic systems.

What is WebClient in Spring Boot?

WebClient is like a helper tool in Spring Boot that helps you make HTTP requests to other web services or APIs.

It’s non-blocking, meaning it doesn’t make your application wait for a response. This makes it faster and better for high-performance apps.

How Event Loop Knows API Response is Completed (WITHOUT Waiting Thread)

🧠 First Simple Idea

👉 Thread doesn’t wait → OS tells when response comes

🧩 Full Diagram + Flow

✅ Step-by-Step Explanation

1️⃣ Request Comes

EventLoop thread picks the request. 

eventLoop-1 handles request
2️⃣ 3 Services Called in Parallel

Requests are sent to all services.

  • Requests are sent ✅
  • Thread does NOT wait ❌

Instead, it says: 

"OS, please inform me when response comes"
3️⃣ Thread Becomes FREE

This is the magic of non-blocking programming. 

eventLoop-1 → now free to handle other requests
  • No blocking
  • No waiting
  • No sleeping
4️⃣ OS Watches Everything

OS continuously checks sockets using: 

epoll (Linux)
kqueue (Mac)

OS checks: 

"Which socket got data?"
5️⃣ Service2 Responds First

Suppose Service2 sends response first. 

Service2 socket → READY ✅

OS sends notification to Event Loop.

6️⃣ Event Loop Reacts
eventLoop-1:
"Oh Service2 response came!"

Event loop executes callback logic.

7️⃣ Same Process for Other Services
  • Service1 done → event loop processes
  • Service3 done → event loop processes
8️⃣ Mono.zip Combines Everything

Important:

  • Not thread waiting
  • Logical waiting
Need all responses:

service1 ✅

service2 ✅

service3 ✅

When all responses complete: 

Combine → Send final response to client ✅

🔥 Important Interview Point

Thread is NOT continuously checking responses.

OS notifies the Event Loop whenever socket data becomes available.

This is called:

  • Event-driven architecture
  • Non-blocking I/O
  • Reactive programming

How to Use and Configure WebClient

1. Add a Dependency

First, you need to add a piece of code to your project so that Spring Boot knows you want to use WebClient. This is done by adding a dependency in your project’s file (like pom.xml if you're using Maven): 

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webflux</artifactId>
</dependency>

2. Create a WebClient Bean

Next, you need to create a WebClient object that you can use in your code. In Spring Boot, we can set this up once and reuse it by creating a “bean.” 

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.reactive.function.client.WebClient;

@Configuration
public class WebClientConfig {

    @Bean
    public WebClient webClient(WebClient.Builder builder) {
        return builder.build();
    }
}

3. Using WebClient to Make Requests

Now that you’ve set it up, let’s see how you can use WebClient to send a request. For example, if you want to get data from https://api.example.com/data, you can do it like this: 

import org.springframework.stereotype.Service;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Mono;

@Service
public class ApiService {

    private final WebClient webClient;

    public ApiService(WebClient webClient) {
        this.webClient = webClient;
    }

    public Mono<String> fetchData() {
        return webClient
                .get()  // Making a GET request
                .uri(uri, accountNumber)  // The URL you want to fetch data from
                .header(Header.PREFER.getKey(), prefareHeader)
                .contentType(MediaType.AAPLICATION_JSON)
                .retrieve()  // Send the request
                .onStatus(
                    HttpStatusCode::isError,
                    response -> response.bodyToMono(Account.class)
                    .flatMap(errorBody -> errorService.failurCause(response, errorBody.getContext(),errorMsg,log))
                )
                .bodyToMono(Account.class);  // Get the response as a string
    }

    public Mono<String> postData() {
        return webClient
                .post()  // Making a POST request
                .uri(uri, accountNumber)  // The URL you want to fetch data from
                .header(Header.PREFER.getKey(), prefareHeader)
                .contentType(MediaType.AAPLICATION_JSON)
                .bodyValue(requestBody)
                .retrieve()  // Send the request
                .onStatus(
                    HttpStatusCode::isError,
                    response -> response.bodyToMono(Account.class)
                    .flatMap(errorBody -> errorService.failurCause(response, errorBody.getContext(),errorMsg,log))
                )
                .bodyToMono(Account.class);  // Get the response as a string
    }
}

4. Getting the Response

Because WebClient is reactive, it doesn’t immediately return the data. Instead, it gives back something called a Mono, which is like a "promise" that data will come later.

To get the data, you can subscribe to it like this: 

apiService.fetchData().subscribe(response -> {
    System.out.println("Response: " + response);
});

Key Points to Remember:

  • WebClient is a new and better way to send HTTP requests in Spring Boot.
  • It can handle both simple and complex tasks, like calling APIs.
  • You don’t need to wait for responses (non-blocking).
  • It is part of Spring WebFlux, which is designed for high-performance applications.
