Part 1: From Scratch – Thread & Runnable

What is a Thread?

A thread is like a worker in your application—it executes a sequence of instructions.

  • The main thread starts your program.
  • You can create additional threads to perform tasks in parallel.
Runnable (task) vs Thread (worker)
  • Runnable → defines what to do (the task)
  • Thread → defines who does it (the worker)

    // Runnable → the work
Runnable task = () -> {
    System.out.println("I am working in: " + Thread.currentThread().getName());
};

// Thread → the worker
Thread t = new Thread(task);
t.start(); // runs task asynchronously
Good things about raw Threads
  • Easy to understand
  • True parallelism (across CPU cores)
  • Great for simple, one-off async tasks
Problems with raw Threads
  • Creation overhead: new Thread() is expensive (memory + OS resources)
  • No pooling: Threads are created/destroyed repeatedly → wasteful
  • No queuing/backpressure: 10,000 requests = 10,000 threads ❌
  • No lifecycle control: Hard to shut down gracefully
  • Error handling is messy: Exceptions inside threads are easy to miss
  • Coordination is painful: Waiting for multiple threads → manual join() chaos
👉 Takeaway: We need a manager of threads → ExecutorService

Part 2: ExecutorService (Thread Manager)

Idea
  • Put tasks in a queue
  • A pool of threads executes them

    ExecutorService pool = Executors.newFixedThreadPool(5);

for (int i = 0; i < 10; i++) {
    int id = i;

    pool.submit(() -> {
        System.out.println("Task " + id + " by " + Thread.currentThread().getName());
    });
}

pool.shutdown();
What problems it solves
  • Thread reuse (pooling) → faster, cheaper
  • Queuing → tasks wait instead of creating too many threads
  • Sizing control → limit max concurrency
  • Lifecycle management → shutdown(), awaitTermination()
Problems / Behaviors
A) execute(Runnable)
  • No Future is returned

    pool.execute(() -> {
    throw new RuntimeException("Boom in execute()");
});
  • If an exception occurs:
  • It goes to the thread’s UncaughtExceptionHandler
  • You will see a stack trace in logs/console (default behavior)
B) submit(Runnable/Callable)
  • Returns a Future

    Future f = pool.submit(() -> {
    throw new RuntimeException("Boom in submit()");
});

try {
    f.get(); // <-- Exception appears here
} catch (ExecutionException e) {
    e.getCause().printStackTrace();
}
  • If an exception occurs:
  • It is captured inside the Future
  • Nothing is shown unless you call get()
Why this happens
  • submit() is designed for deferred error reporting via Future
  • If you don’t call get(), the error is invisible
Why this is a real problem in production
  • Teams often use submit() in fire-and-forget style
  • They don’t store the Future
  • Exceptions never appear in logs
  • System looks “fine” but tasks silently fail
  • Missed notifications
  • Partial processing
  • Data inconsistency
This is called the Error Visibility Problem
Other limitations
  • No method chaining
  • Difficult to trigger next task after completion
  • Mixing async + blocking (get()) makes it inefficient

Part 3: CompletableFuture (Modern Async Pipelines)

Idea

A Future that you can complete, chain, and handle without blocking

Two ways to start tasks

    // Fire-and-forget (no return value)
CompletableFuture<Void> r =
    CompletableFuture.runAsync(() -> sendEmail());

// Compute-and-return (with value)
CompletableFuture<String> f =
    CompletableFuture.supplyAsync(() -> fetchUser());
Chaining (no blocking)

    CompletableFuture.supplyAsync(() -> fetchUser())
    .thenApply(user -> enrich(user))
    .thenCompose(u -> saveAsync(u))
    .thenAccept(saved -> System.out.println(saved))
    .exceptionally(ex -> {
        log.error("Failed", ex);
        return null;
    });
Parallel composition (real speed-up)

Call multiple services in parallel and combine results:


    CompletableFuture<Product> p =
    CompletableFuture.supplyAsync(() -> getProduct());

CompletableFuture<Pricing> pr =
    CompletableFuture.supplyAsync(() -> getPricing());

CompletableFuture<Reviews> r =
    CompletableFuture.supplyAsync(() -> getReviews());

CompletableFuture<Combined> result =
    p.thenCombine(pr, (pp, price) -> new Pair<>(pp, price))
     .thenCombine(r, (pair, rev) ->
         new Combined(pair.first, pair.second, rev));

Combined value = result.join();
What problems it solves vs ExecutorService + Future
  • No need to call get() early → keeps flow async
  • Supports chaining: thenApply, thenCompose, thenCombine
  • Clean error handling: exceptionally, handle, whenComplete
  • Better for microservice composition
Still not perfect (honest reality)
  • If you use blocking calls (e.g., JDBC, RestTemplate) inside supplyAsync:
  • Thread is still blocked
  • This becomes Async + Blocking
  • Overusing join() / get() makes it synchronous again
  • Carefully size thread pools
  • Separate CPU vs I/O pools
Final Note

For extreme concurrency + resource efficiency, move to Reactive Programming (WebClient / Project Reactor):

  • Uses non-blocking I/O
  • Uses event loops instead of threads

Real-Time Banking Scenario: Multi-Threading

Money Transfer (NEFT / IMPS / RTGS) – Parallel Validations
Scenario

A customer transfers ₹50,000 from Account A to Account B.

Before money moves, bank must do multiple checks:

  • Balance check
  • Fraud check
  • Account status check
  • Daily limit check
Without Multi-Threading (Sequential)
  • Balance check → 100 ms
  • Fraud check → 200 ms
  • Account status → 100 ms
  • Limit check → 100 ms

Total = 500 ms

User feels delay ❌

With Multi-Threading (Parallel)

All checks are independent, so run in parallel:

  • Balance check
  • Fraud check
  • Account status
  • Limit check

Total time = max(200 ms)

Fast response ⚡

💡 Insight: Parallel execution reduces latency dramatically in banking systems.
