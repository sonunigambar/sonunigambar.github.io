❌ Without Messaging System (Bad Architecture)

In this architecture, every service directly communicates with every other system. This creates a highly complex and tightly coupled system.

🚨 Problems:
  • Tight coupling between services
  • Too many connections (N × M problem)
  • Hard to maintain and scale
  • No fault tolerance (data loss risk)
  • Every new service requires changes everywhere

✅ With Messaging System (Good Architecture)

Here, all services communicate through a central messaging system. This removes direct dependencies and enables scalable, event-driven architecture.

🔥 Advantages:
  • Loose coupling between services
  • Supports multiple consumers (Pub/Sub model)
  • Highly scalable and flexible
  • Fault tolerant (messages are stored)
  • Easy to add new services without changing existing ones

📩 Types of Messaging Systems

❌ Point-to-Point Messaging

In this model, a message is sent to a queue and only one consumer receives it.

Producer → Queue → One Consumer
Key Points:
  • One message → One consumer
  • Used for task processing
  • Load is distributed
Limitation:
  • Cannot send message to multiple services

✅ Publish-Subscribe (Pub/Sub)

In this model, a message is sent to a topic and multiple consumers receive it.

Producer → Topic → Multiple Consumers
Key Points:
  • One message → Many consumers
  • Event-driven architecture
  • Highly scalable
Advantages:
  • Loose coupling
  • Easy to add new services
  • Supports real-time processing

🍔 Kafka & Zookeeper Analogy (McDonald's Example)

Understanding Kafka becomes easier with a real-world analogy.

🏬 McDonald's Company has many outlets
Each McDonald's outlet:
  • Takes customer orders 🍔
  • Prepares and stores the food
  • Delivers the food to customers 🍟
  • Each outlet knows only about its own kitchen
🏢 McDonald's Headquarters:
  • Keeps a list of all working outlets 🏪
  • If an outlet goes down, it reassigns work to others
  • Helps coordinate food delivery and team management
⚙️ What is Zookeeper?

Zookeeper acts like the headquarters of the system. It manages and coordinates all Kafka brokers.

  • Keeps track of all active Kafka brokers
  • Performs leader election for partitions
  • Handles broker failures and recovery
  • Maintains cluster metadata

Just like McDonald's headquarters ensures all outlets are working properly and assigns work when one fails, Zookeeper ensures Kafka cluster stays healthy and consistent.

🔁 Kafka Mapping
  • Outlets → Kafka Brokers
  • Orders → Messages
  • Customers → Producers / Consumers
  • Headquarters → Zookeeper

📩 Kafka Events (Key, Value, Timestamp)

In Kafka, we don't just send simple messages. Instead, we send events. An event represents something that happened in the system.

📌 What is an Event?

An event in Kafka contains the following:

  • Key: Identifies the event (used for partitioning)
  • Value: Actual data (business information)
  • Timestamp: When the event occurred
🧠 Example Event
Key: order-123
Value: Order Created with amount ₹500
Timestamp: 2026-05-03T22:30:00
🔥 Why Key is Important?

Kafka uses the key to decide which partition the event will go to. All events with the same key go to the same partition, which ensures ordering.

👨‍💼 Real Example (Employee Processing)

Suppose employee events are coming continuously, and you want to process them department-wise in order. 


HR: Employee A joined

HR: Employee B promoted

IT: Employee C joined

IT: Employee D resigned

If you use department as key:

  • All HR events go to same partition → processed in order ✅
  • All IT events go to another partition → processed in order ✅

✔️ This ensures correct sequencing within each department.

❌ Without key → events may go to different partitions → order can break.

📤 Sending Event (Producer)

bin/kafka-console-producer.sh \

--topic myLog \

--bootstrap-server localhost:9092 \

--property "parse.key=true" \

--property "key.separator=:"

Send events:

HR:Employee B promoted

IT:Employee C joined

bin/kafka-console-consumer.sh \

📂 Kafka Topic, Partition & Offset (Simple Explanation)

Let's understand Kafka concepts using a simple folder analogy.

📁 Topic (Folder)

A topic is like a folder where all related messages (events) are stored.

📂 Partition (Subfolder)

Each topic is divided into multiple partitions (subfolders).

  • Each partition stores part of the data
  • Helps in parallel processing
  • Improves scalability
📄 Messages (Files)

Messages inside partitions are like files inside subfolders.

🔢 Offset (File Number)

Each message has a unique offset (ID) within a partition. 

Offset 0 → Message 1
Offset 1 → Message 2
Offset 2 → Message 3
⚙️ Partition Assignment Logic
  • If key is present → same key goes to same partition
  • If no key → messages distributed evenly (round-robin)
👑 Leader & Follower
  • Leader: Handles all reads and writes
  • Followers: Replicate data for reliability
  • If leader fails → follower becomes leader

❗ Ordering is guaranteed only within a partition, not across the entire topic.

📊 Kafka Partition, Log & Segment (Based on Diagram)

📌 Topic & Partitions

A Kafka topic is divided into multiple partitions (P0, P1, P2). Each partition works independently and allows parallel processing.

📂 Inside Partition (Log Structure)

Each partition stores data as an append-only log. 

Partition → Log → Segments → Messages
  • Partition contains a log
  • Log is divided into segments
  • Segments store messages (events)

Kafka creates new segments based on size or time.

🔢 Offset (Message Position)

Each message in a partition has a unique offset (position). 


Offset 0 → Employee A joined  

Offset 1 → Employee B promoted  

Offset 2 → Employee C resigned
  • Offset starts from 0 and increases
  • Unique only within a partition
  • Used by consumers to track progress

✔️ Consumer reads messages using offset and can resume from last position

⚙️ Partition Selection Logic

Kafka calculates partition using the message key: 

partition = hash(key) % number_of_partitions
🧠 Example
Key Hash % 3 Partition
HR 0 P0
IT 1 P1
HR 0 P0 ✅

✔️ Same key always goes to same partition → maintains ordering

❗ Without key → messages distributed round-robin way → ordering not guaranteed

📊 Kafka Leader, Follower & Replication Factor

📌 Topic Creation
bin/kafka-topics.sh --create \
--topic myLog \
--bootstrap-server localhost:9092 \
--partitions 3 \
--replication-factor 3

Here we created a topic myLog with:

  • 3 Partitions → P0, P1, P2
  • Replication Factor = 3 → Each partition has 3 copies
🔁 Replication Factor

Replication factor means how many copies of each partition are stored across brokers.

✔️ More replicas → More reliability and fault tolerance

👑 Leader & Follower

Each partition has:

  • 1 Leader → Handles all read and write operations
  • Followers → Copy data from leader

Example:

  • P0 → Broker-101 (Leader)
  • P0 → Broker-102 (Follower)
  • P0 → Broker-103 (Follower)
🔄 How It Works
  • 1️⃣ Producer always sends data to the Leader
  • 2️⃣ Followers replicate data from the Leader
  • 3️⃣ Consumer reads data from the Leader
⚠️ What Happens if Leader Fails?

If the leader goes down:

  • One follower becomes the new leader
  • System continues without data loss

✔️ This is why Kafka is highly available

❗ Clients always write to leader, followers never accept direct writes
